INSTALLATION

  • Run the Install self extracting executable
  • rf_v<version>.zip and rf_gpp_v<version>.tar.gz will be installed in your CCS IDE installation directory e.g. c:\ti. 
  • rf_v<version>.zip: DSP-side source code for RF Levels 1, 3, 5, 6 and associated framework modules
  • rf_gpp_v<version>.tar.gz: GPP-side minimal source code for RF Level 6 application and associated modules.
  • After installation, extract the rf_v<version>.zip into any directory to obtain the source code.

Refer to SPRA79x, (x = framework level number), under Chapter 3 "Installing and Running the RF x Software" for details on how to build and run the application.

  • If using RF6, please refer to the chapter 4, Installing, Building, and Running the RF6 Base Application in spra796 for details on how to build and run the application. To extract the tar.gz, on the Linux workstation: -
  1. Make a directory with name of your choice
    [>] cd ~/montavista
    [>] mkdir -p xyzdir
  2. Go into the directory
    [>] cd xyzdir
  3. Untar the GPP-side code in a directory of your choice:
    [>] tar xvzf rf_gpp_v<version>.tar.gz

CCS Update steps (for DSK 6713 and DSK 6416 only)

RF v3.1 content was built with CCS 2.20.18 to leverage the latest emulation, DSP/BIOS etc. for the DSK6416 and DSK6713 boards. This may be newer than your DSK-only standalone software. Since DSK-only customers are not presently able to leverage CCS serial updates (such as CCS 2.20.18) some additional steps may be required to build and run Target Content.
  • Standalone DSK6416 and DSK6713 customers
  1. Open an MS-DOS command prompt and run dosrun.bat from the CCS for your DSK installation (e.g. c:\ti\)
  2. Run the top-level build configuration script referenceframeworks\buildConfig.bat. This will create DSP/BIOS configuration files (cdb's) compatible with your DSK release, for all RF levels. Alternatively just run makeConfig.bat in the individual RF Level board port of interest (e.g. c:\referenceframeworks\apps\rf3\dsk6416). 
  3. Start CCS, open the RF project for your board (e.g. c:\referenceframeworks\apps\rf3\dsk6416\app.pjt)
  4. Go to Project -> Build Options. Add the following include path: -i"$(Install_dir)\c6000\xdais\include" (see SDSsq31521)
  5. Rebuild and run the application
  • DSK6416 and DSK6713 customers with full CCS licenses
  1. Install CCS version 2.20
  2. Install DSK 6416 update for CCS 2.2 (CD provided with DSK 6416 board) or DSK 6713 update depending on which board you have. If you have both boards install the DSK6416 update first.
  3. Install CCS version 2.20.18 (available via Update Advisor).
  4. Rebuild and run the application

Note:  The DSK CD patches will not install on top of CCS version 2.20.18 

  • All others customers should install CCS v2.20.18 or higher (available via Update Advisor) to build and run RF v2.2 applications.  

Note : Customers with CCS 2.21 can directly rebuild and run RF v2.2 applications since CCS 2.21 uses the same DSP/BIOS version as CCS 2.20.18     

  • Open CCS 2.20.18 (or higher) for a supported target. Build and run the application code.

SDSsq36175 DSP-side AIC23 MCBSP IOM codec driver uses reg-level CSL 3.x calls Ref Framework 6 OMAP
Release Note 
RF v3.01's IOM codec driver uses CSL 3.x register level calls for 2 
things :-
(i) set the ENDPROG bit via a CSL_FINS call in configDma() to indicate 
end of programming of next packet address, size.
(ii) to ping the CSR thus clearing interrupts and making way for 
additional interrupts. In fact no reg level calls are made here but we 
need to talk to low level structure members...which are effectively at 
register level.
This needs to be fixed in a future release since the RF6 IOM codec 
driver should only be relying on Functional Layer API calls to shield 
from lower-level details (which may possibly change)

SDSsq33302 RF 6 provides an additional header to be use with CSL V3.x Ref Framework 6 OMAP
Release Note 
An additional header file (stdplus.h) has been provided in the 
Reference Frameworks release under the public 
referenceframeworks\include folder.
This is to temporarily work around the lack of sized types in DSP/BIOS. 
For example, it is the future intention of DSP/BIOS to use e.g. Uint16 
instead of MdUns, thus making data size explicit. Additionally we 
desire that all components (DSP/BIOS, CSL etc) use the same basic 
types, rather than introduce new ones. 
CSL v3.x therefore includes a file tistdtypes.h which gathers all the 
primitive "C" data types into one consolidated file. This file is 
included by csl.h. This file does *not* specify the types
for aggregate data such as "C" structures or vectored quantities.
All the types defined in this header are bracketed with _TI_STD_TYPES
symbol definition scope as a means of protection against multiple 
definitions. For example :-
#ifndef _TI_STD_TYPES
#define _TI_STD_TYPES
...typedef definitions...
#endif /* _TI_STD_TYPES */
However, the version of DSP/BIOS in CCS 2.20.18 does not yet have this 
data type schema in place. 
Hence an additional file stdplus.h was added to ensure that components 
(e.g. IOM drivers) could seamlessly use std.h and csl.h, and 
correspondingly both 'original' DSP/BIOS types and 'new' sized types. 
stdplus.h prevents build warnings. The stdplus.h file needs to be 
included before std.h and csl.h if using CSL v3.x
Note that when a new DSP/BIOS std.h with sized types support is 
released, existing driver code which #include's stdplus.h will still 
build/run fine. This is ensured because there is agreement on the usage 
of the multiple inclusion pre-qualifier _TI_STD_TYPES. stdplus.h will 
simply become redundant at that time.

SDSsq33300 RF6 must add source file to change IOM driver's default structures Ref Framework 6 OMAP
Release Note

The Reference Framework 6 application uses the default device parameter structure set in the IOM device driver. The device driver default values are DMA channel 4 and 5 and McBsp 1 which is connected to the AIC23 audio codec, on the Innovator5910 platform..

There are two ways to change DMA channels:

1) Static way:
-----------

Example: Default parameters and different DMA channels.

INNOV5910_DMA_AIC23_DevParams myParams = {
    INNOV5910_DMA_AIC23_VERSION_2
    2,                                                     // DMA receive channel
    3,                                                     // DMA transmit channel
    INNOV5910_DMA_AIC23_PORTTYPE_DEFAULT,
    {
        INNOV5910_DMA_AIC23_IER_MASK_DEFAULT,
        INNOV5910_DMA_AIC23_IER_MASK_DEFAULT
    },
    {
        INNOV5910_DMA_AIC23_IER_MASK_DEFAULT,
        INNOV5910_DMA_AIC23_IER_MASK_DEFAULT
    }
};

Pass "_myParams" to the User-Defined Device parameter.

2) Dynamic way:
------------
You need to define a function which will just change some particular fields. This function will be implemented in another file called initParams.c which will be added into your application.

Example: Default parameters and different DMA channels.

INNOV5910_DMA_AIC23_DevParams myParams =
    INNOV5910_DMA_AIC23_DEVPARAMS_DEFAULT;

void initDevParams()
{
    myParams.rxDmaId = 2;
    myParams.txDmaId = 3;
}

Within DSP/BIOS Configuration Tools:
- Pass "_myParams" to the User-Defined Device parameter.
- Pass "_initDevParams" to the User Init Function from the Global
Settings.

This example shows how to use DMA channel 2 and 3. DMA channel 2 and 3 will be plugged into the correct interrupt.

SDSsq32251 Incorrect assertion for SIO_issue in RF5 Ref Framework 5 All
Release Note

In RxSplit.c and TxJoin.c, the startup functions use UTL_assert to check the return code rc of SIO_issue: rc = SIO_issue(...); UTL_assert( rc >= 0);

This assertion is incorrect because SIO_issue returns SYS_OK if successful and some constant corresponding to a positive integer for failures. Hence UTL_assert will never assert, even on failed SIO_issue calls.

The above call should be replaced with: UTL_assert (rc == SYS_OK);

SDSsq29946 Reference Frameworks failed to build with C++ compiler All Ref Frameworks All
Release Note

When the C++ compiler is enabled, the RFs build with an error for the 
file *cfg_c.c file, saying "error: badly formed pragma". This is 
because the syntax of the CODE_SECTION pragma used in the file for 
CSL_cfgInit is incompatible with C++. C++ supports a different syntax 
where the pragma has to be directly located before the function it 
addresses. The workaround here is to right click on *cfg_c.c in the 
Generated Files of the project view, select Files Specific Options and 
uncheck "Treat C files as C++ files" under Parser of the Compiler Tab.

SDSsq26186 RF3 GEL sliders behave differently across board ports Ref Framework 3 All
Release Note

On the Dsk5416 and Teb6416, when the set_active_channel slider is changed from 0 to 1 and vice versa, sometimes the Left/Right output does not toggle. Similar inconsistency problems occur with the gain sliders. Volume increases or decreases, but the audio toggles from one channel to the other.

This is an unsolved problem so far on anything using the PCM3002 codec. 
It may not be considered too serious, as it only happens in debug 
(halt) mode. When the user halts the target, for example by hitting a 
breakpoint or just halting it manually, if he halts it when the second 
sample of the two-sample frame (left + right) was being
transmitted, that one will complete and everything will be fine. But if
the user halts it when the first one was transmitted, the left sample 
will complete and silence will be sent to the right. But when the user 
resumes, the remainder of the frame, i.e. the second sample, is now 
sent to the left channel and from that point on the channels are 
reversed. Only if the user halts it again at the first word and resume, 
things will flip back to normal.

When changing anything using GEL files, the target is really halted, 
the memory changed, and the target is resumed. So there is a 50% chance 
of channels flipping every time the GEL file is used.

This problem only applies to the PCM3002 codec.

SDSsq25479 GEL sliders on RF3 DSK6711 cause excessive breakpts and halt codec Ref Framework 3 C6000
Release Note

GEL sliders on RF3 DSK6711 cause excessive breakpts and halt codec. RF3 
has a few sliders to do exercise minimal application control. However, 
CCS stops the target on each increment of a slider - i.e. it does not 
just halt the target at the end-point - it halts it at every change on 
the way. Hence the number of breakpoints can be large. The DSK6711 codec 
(AD535) has a known issue that when excessive breakpoints occur the 
codec LSB can be set erroneously, causing codec to interpret it as a 
command control word and sending it into the weeds!

One workaround for the user is to decrease the granularity of slider 
movements. The current increment is 10, with range of 0 to 200. The 
app.gel source is available in RF's to do this.

SDSsq36168 issues putting .bios in external memory with Link & RF6 on 5912 Ref Framework 6 OMAP
Release Note 
RF v3.01 DSP-side is shipped by default with :-
.text:init
.text:create
.text:delete
.text:exit
.rtdx_text
in EXTMEM (external memory...SDRAM). ie non-critical code 
in external memory. All runtime .text is still in fast onchip
SARAM
.bios 
in SARAM (internal memory...single-access RAM)
It has been observed that with .bios in EXTMEM, the DSPLink loopgpp
example fails on 5912. This issue is still being investigated.
However, users can safely put .text in EXTMEM thus opening up 
a large memory map for users.
Users should keep .bios in internal memory until this issue has been 
resolved.

Fix Description

Root cause was as noted in DDTS 36205.

Erroneous setting of GBL.CLKOUT on DSP-side of 12Mhz (instead of 
192Mhz) was causing CLK timer interrupts to go off at a very high 
rate. When .bios was moved to external memory, there was not enough 
MIPS to run the DSP.

Simple fix was to ensure GBL.CLKOUT = 192.0 in the Tconf script.
